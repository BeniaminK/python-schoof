Detailed Description

Class decorators for more expressive call profiles.

The Python profiler in the cProfile module registers methods as elements of the class that contains their source code. Thus, cProfile merges calling statistics of the defining class and subclasses that do not re-implement the methods (such as template class specializations). The decorator local_method_names() addresses this problem. Also, in case of nested template classes (say, elliptic_curves.polynomials.naive.CurvePolynomials), the template parameters bloat the type name; the decorator profiling_name() alleviates the issue.

def support.profiling.local_method_names (   cls)

A class decorator that makes the function names used by the cProfile module local to the class of use (rather than the class of definition).

The cProfile module uses the name of a function's code object as the profile name. Therefore, calls to methods in subclasses that do not re-implement the method are counted as calls to the parent class. In template classes, this makes the call profile too coarse.

Use profiling_name() to get shorter class names.

Note
The decorator copies the code objects; it "flattens" the class. Therefore, re-assigning methods will annihilate the decorator effects for the method.
See Also
The sources of rings.quotients.naive.QuotientRing for a usage example and support.types.template() for information about template classes.

def support.profiling.profiling_name (   name)

A class decorator that sets the name that will show up in profiles generated with the cProfile module.

Usage example:

1 @local_method_names
2 @profiling_name( "GF<{_size}>" )
3 class FiniteField( Field, metaclass=template( "_size" ) ):
4  ...
Parameters
nameA string that contains the class name (which is usually shorter than the original name). For template classes, the string may contain format-string variables for the parameters.
Note
The function requires the local_method_name() decorator to show an effect.
