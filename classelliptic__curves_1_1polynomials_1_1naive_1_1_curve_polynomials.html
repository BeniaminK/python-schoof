Detailed Description

A ring of polynomials over an elliptic curve; the polynomials support infix notation for ring operations, as well as mixed argument types.

This is a template class that must be instantiated with the elliptic curve. Use it, for example, as follows:

1 # Create the underlying elliptic curve
2 E = elliptic_curves.naive.EllipticCurve( FiniteField(7), 3, 4 )
3 # Instantiate the template; GF7E is a class (here: polynomials over E)
4 GF7E = CurvePolynomials( E )
5 
6 # Use two coefficient lists as arguments; the polynomial is (list1) + y*(list2)
7 s = GF7E( (4, 3, 0, 1) ) # The curve's defining polynomial
8 y = GF7E( 0, (0, 1) ) # Another polynomial: y
9 y**2 == s # The fundamental relation holds
10 type(s) is GF7E # This is also true
11 
12 # Use the polynomials 'x' and 'y' for a more natural syntax
13 x = GF7( (0, 1), () ) # The empty tuple is just for emphasis
14 y = GF7( (), (0, 1) ) # The empty tuple is just for emphasis
15 y**2 == x**3 + 3**x + 4
16 p = x**2 + 2* x + y*( 3 * x**5 + 6 * x**2 ) # A (pseudo) random polynomial

The ring of polynomials over an elliptic curve is , where is the field over which the curve was defined, and are the curve parameters. The quotient polynomial expresses that the polynomial's domain is the set of points on ; the (Cartesian) coordinates of the points are connected by the fundamental relation .

Note
As a consequence of the fundamental relation, powers of greater than one can be reduced to a polynomial in . This is what this implementation does. Since it does not provide full multivariate arithmetic, division (with remainder; see __divmod__()) only works in the following cases:
  • The divisor is a polynomial in alone; this works for all dividends.
  • Dividend and divisor are polynomials in alone.

Besides restricting arithmetic, the implementation emphasizes simplicity over speed; it omits possible optimizations.

Note
The class uses the operand_casting() decorator: other operands in binary operations will first be treated as CurvePolynomial elements. If that fails, the operation will be repeated after other was fed to the constructor __init__(). If that fails, too, then the operation returns NotImplemented (so that other.__rop__() might be invoked).
